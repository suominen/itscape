# ITScape Tracking — Claude Code Context

This repository contains a living tracking document for **ITScape**
(**CVE-2026-46316**), a KVM/arm64 guest-to-host escape in the Linux kernel.
A translation-cache double-put in `vgic_its_invalidate_cache()` —
it puts the *iterated* pointer instead of the value returned by
`xa_erase()`, and the function runs from mutually-non-exclusive contexts
(`its_lock`, `cmd_lock`, and the `GICR_CTLR` `EnableLPIs` path that holds
neither) — drops a `vgic_irq`'s single cache reference more than once under
concurrency, freeing it while an ITE still maps it: a **use-after-free** a
guest with EL1 can drive to host-kernel code execution.

The canonical fix is [`13031fb6b835`][fix] (*KVM: arm64: vgic-its: Drop the
translation cache reference only for the erased entry*), first released in
**v7.1**. The bug was introduced by `8201d1028caa` in **v6.10**
(2024-04-25), so the reachable window is **6.10 through 7.0**; **kernels
older than 6.10 are not affected**.

**CVE-2026-46316** is assigned (CVSS 3.1 8.8,
`AV:L/AC:L/PR:N/UI:N/S:C/C:H/I:H/A:H`). Discovered by Hyunwoo Kim
(`@v4bel`) and disclosed 2026-07 — the first known guest-to-host escape for
KVM/arm64. PoC: <https://github.com/V4bel/ITScape>.

ITScape is **arm64-only**. Its **x86 sibling** by the same researcher is
**Januscape (CVE-2026-53359)** — a KVM/x86 shadow-MMU escape, tracked
separately at <https://kimmo.cloud/januscape/> (repo `~/src/januscape`).
Keep the two cross-referenced.

The rendered site is published at <https://kimmo.cloud/itscape/>.

[fix]: https://github.com/torvalds/linux/commit/13031fb6b8357fbbcded2a7f4cba73e4781ee594

## What flips a verdict — the kernel backport, and only that

This bug lives entirely in the kernel and there is no runtime knob to
disable the vGIC-ITS path (a guest needs its interrupts). A row's verdict
is one question:

- **Kernel** — is the **arm64** kernel inside the reachable window
  (6.10–7.0) *and* missing the [`13031fb6b835`][fix] backport? A kernel
  **< 6.10** never had the buggy rework and is **not affected**; a kernel
  carrying the fix is patched. This is what *Status* and *Fixed since* key
  on.

The narrow 6.10 origin means many stable EL and Debian releases are
genuinely **not affected** (contrast the x86 Januscape sibling, whose
2.6.36 origin puts every kernel in-window). Triggering needs guest
**root/EL1**; note that as an exploitation caveat in prose, not a column.

The combined distribution-status table is the **single source** for each
row's kernel, *Fixed since*, and status. Don't restate the table's columns
in prose or add a parallel per-release table. The **Kernel** cell holds the
version only (or `:grey_question:` when unverified). Label NixOS channels in
the **Release** column in friendly form (`Unstable`, `26.05`). Keep the
`{.distros}` block attribute on the line **immediately after** the table.

Track the **arm64** build of each distro. **Proxmox VE is x86-only** — it
does not appear in this tracker (it is covered by Januscape); do not add a
Proxmox row here.

## Routine run scope — frozen version snapshots

The per-distro **Kernel** versions in the table are **frozen seed
snapshots**, not live values: update a row's recorded version only when its
verdict actually changes. A row cannot flip to fixed until the fix is
available to adopt — its kernel series carries :white_check_mark: in the
*Upstream fixed versions* table, **or** the distro cherry-picks
`13031fb6b835` independently.

Each run:

- Re-check the *Upstream fixed versions* table: every maintained in-window
  branch (6.12.y, 6.18.y, 7.0.y, 7.1.y) already carries the fix (6.12.93 /
  6.18.35 / 7.0.12 / 7.1). Pre-6.10 branches (6.6.y and older) are **not
  affected** — leave them `:heavy_minus_sign:`.
- For a distro row, re-pull the distro's **arm64 kernel** version and
  compare: a kernel < 6.10 stays *not affected*; an in-window kernel that
  reaches its branch's first-fixed release (or ships the cherry-pick) flips
  to :white_check_mark:, record the adopted kernel, set *Fixed since*.
- A version bump that stays **inside the window without the backport** is
  **not** adoption ⇒ leave the row untouched.
- Watch AlmaLinux (leading indicator) for the EL10 arm64 row.

`zcat` / `gunzip` **are** in the headless allowlist — use them for the
repodata pulls. Pull only the kernel version, only when a verdict could
change.

**Never record NixOS channel git-revisions** in the tracker — they advance
on nearly every run.

## Conventions for status entries

- `:white_check_mark:` — either the release's kernel carries the
  `13031fb6b835` backport (record the fixed kernel version and set *Fixed
  since*) — **or** the kernel is outside the reachable window (**< 6.10**),
  i.e. *not affected*; note the reason ("predates the trigger (< 6.10)").
- `:x:` — vulnerable: an in-window (6.10–7.0) arm64 kernel without the
  backport.
- `:warning:` — the fix is staged but not yet in the user-facing channel
  (merged / cherry-picked but not released). A mitigation is not a fix.
- `:grey_question:` — not yet verified (kernel pin or advisory not yet
  inspected).
- `:heavy_minus_sign:` — used in the *Upstream fixed versions* table for a
  stable branch that predates the reachable window (6.6.y and older).

### "Fixed since" column

`Fixed since` is a **sticky first-observation** date: the date the
release's kernel fix first held. Set it when a row flips to fixed; only bump
it if the verdict flips again or the fixed kernel version changes. Leave it
`—` while the row is vulnerable or unverified — and for "not affected" rows,
which were never vulnerable.

### Verification log

Update the `## Verification log` section rather than appending a line per
re-check. Bump the `*Last verified <date>.*` line and edit the relevant
`### Upstream` / `### Distributions` subsection in place. The verified-date
lives **only** in that header — do not repeat it inline. Method/source
attribution without a date is fine.

### Date handling — first-seen / last-changed, not "today"

Dates in the prose are **first-seen / last-changed** dates, not "today".
Only move a date when the fact it qualifies actually changes. If the entire
run is a no-op, leave the file alone and don't commit — don't bump
`lastmod`, don't bump `*Last verified*`.

## Auto-update worktree

The systemd timer runs `scripts/auto-update` from a dedicated git worktree
at `~/src/auto-update/itscape`, checked out on a single long-lived branch
named `auto-update`. The wrapper merges `origin/main` forward into that
branch on each run, then hands off to headless Claude, which commits any
tracker changes back onto `auto-update` only. The agent must not create
per-run branches, switch branches, push, or open PRs — merges of
`auto-update` into `main` are done manually by the user.

One-time setup (from the primary checkout at `~/src/itscape`):

```
git worktree add -b auto-update ~/src/auto-update/itscape main
```

The wrapper runs `git fetch origin` and `git merge origin/main` under
`set -e`, so the repo needs a GitHub `origin` with `main` pushed or every
scheduled run aborts before doing anything.  A freshly `git init`'d tracker
must `git remote add origin https://github.com/suominen/itscape.git` and
`git push -u origin main` before the timer is worth enabling.

To run a refresh immediately (same path the timer takes):

```
systemctl --user start itscape-tracker-update.service
```

It is a `oneshot`, so the command blocks until the run finishes; follow its
output with `journalctl --user -u itscape-tracker-update`.  Run the trackers
**one at a time**, never in parallel — they share the `~/src/linux/*`
reference clones.

Wire the timer by symlinking both units into `~/.config/systemd/user/` with
relative links (`ln -sr`) and enabling it:

```
ln -sr ~/src/itscape/systemd/itscape-tracker-update.service \
       ~/src/itscape/systemd/itscape-tracker-update.timer \
       ~/.config/systemd/user/
systemctl --user daemon-reload
systemctl --user enable --now itscape-tracker-update.timer
```

The timer fires at `06,18:05` — staggered from the sibling trackers
(ipv6_frag_escape `:05`, cifswitch `:20`, pintheft `:35`, januscape `:50`) so the shared
kernel clones are not fetched simultaneously.  Verify the live set with
`systemctl --user list-timers | grep tracker` — this in-doc list has gone
stale before.

## Tearing down the auto-update

Unwire in this order — the sequence matters, because `systemctl disable`
needs the unit definition to still be resolvable when it runs.

1. **Disable the timer first**, while the unit symlinks are still in place:

   ```
   systemctl --user disable --now itscape-tracker-update.timer
   ```

2. **Remove the unit-definition symlinks**:

   ```
   rm ~/.config/systemd/user/itscape-tracker-update.timer \
      ~/.config/systemd/user/itscape-tracker-update.service
   ```

3. **Reload** so the running user manager drops the units:

   ```
   systemctl --user daemon-reload
   ```

If the definition symlinks were removed *before* disabling, the stale
`enable` symlink parks the timer in a `failed` state. Recover with:

```
rm ~/.config/systemd/user/timers.target.wants/itscape-tracker-update.timer
systemctl --user daemon-reload
systemctl --user reset-failed itscape-tracker-update.timer
```

Finally, remove the worktree and its branch (run from `~/src/itscape`):

```
git worktree remove ~/src/auto-update/itscape
git branch -d auto-update
```

## Local kernel clones for git.kernel.org

git.kernel.org's cgit HTML pages are Anubis-gated; WebFetch hits the no-JS
challenge. The agent inspects kernel history via long-living local clones
under `~/src/linux/`:

| Clone path           | Upstream                                                           |
|----------------------|--------------------------------------------------------------------|
| `~/src/linux/stable` | `https://git.kernel.org/pub/scm/linux/kernel/git/stable/linux.git` |
| `~/src/linux/kvm`    | `git://git.kernel.org/pub/scm/virt/kvm/kvm.git`                    |
| `~/src/linux/vulns`  | `https://git.kernel.org/pub/scm/linux/security/vulns.git`          |

`kvm` is the KVM maintainer tree, where KVM arm64/x86 fixes land and get
tagged for stable before Linus (shared with the Januscape tracker). `vulns`
is the kernel CNA's CVE database.

The auto-update *wrapper* refreshes the clones before invoking the headless
agent; the agent inspects via the `origin/<branch>` remote-tracking refs
(e.g. `origin/linux-6.12.y`).

**Is a stable branch fixed yet?**  The backport is a cherry-pick with a new
SHA, so search by upstream reference / subject:

```
git -C ~/src/linux/stable log v<series>..origin/linux-<series>.y --grep=13031fb6b835 --grep='translation cache reference' --format='%h %s'
```

Empty output ⇒ still unpatched. Keep the range bounded to `v<series>..` —
an unbounded subject grep over the whole branch history can match an
ancient unrelated commit and read as a false "fixed". Confirm the fix landed mainline in v7.1
with:

```
git -C ~/src/linux/stable describe --contains 13031fb6b835
```

## The CVE record (`vulns.git`)

**CVE-2026-46316** is already assigned. The kernel CNA's `vulns.git` keys
each record on the *fixing commit SHA*; inspect via `origin/master`:

```
git -C ~/src/linux/vulns show origin/master:cve/published/2026/CVE-2026-46316.dyad
```

The `.dyad` gives the authoritative per-branch `<introduced>:<fixed>`
versions (introduced 6.10 → fixed 6.12.93 / 6.18.35 / 7.0.12 / 7.1); the
`.cvss` file carries the 8.8 vector. Watch NVD / the disclosure sources for
a KEV/EPSS update to add to the Summary.

## Local nixpkgs clone for NixOS channel verification

*Seeding-and-adoption method — see "Routine run scope" above. Do not record
the resolved channel revisions in the tracker.*

NixOS rows are verified from a local nixpkgs clone at `~/src/nixos/nixpkgs`.
Each channel has a git-revision pointer at
`https://channels.nixos.org/<channel>/git-revision`; at that commit,
`pkgs/os-specific/linux/kernel/kernels-org.json` carries the version for
every supported mainline.org kernel series (arm64 uses the same versions).
The default `linuxPackages` is set by `packageAliases.linux_default` in
`pkgs/top-level/linux-kernels.nix` — currently `linux_6_18` (6.18.38), with
`linux_latest` = `linux_7_1` (7.1.3); both carry the backport. Read that
alias rather than assuming the default is the newest or oldest LTS — it has
moved before.

Tracked channels: `nixos-26.05` (default), `nixos-unstable`.

```
rev=$(curl -fsSL https://channels.nixos.org/<channel>/git-revision)
git -C ~/src/nixos/nixpkgs show "${rev}:pkgs/os-specific/linux/kernel/kernels-org.json"
```

`channels.nixos.org/<channel>/git-revision` returns a 302 — always pass `-L`
to curl. The wrapper refreshes the clone on every run.

## Rocky / Amazon kernel version source (RPM repodata)

*Seeding-and-adoption method — see "Routine run scope" above. These indexes
are gzipped; `zcat` is in the headless allowlist. Track the **aarch64**
repos.*

Both ship `kernel` as an RPM; pull versions from repodata (`repomd.xml` →
the `*-primary.xml.gz` index) and pick the numerically-highest `rel`
(`sort -V`).

- **Rocky** BaseOS aarch64:
  `https://dl.rockylinux.org/pub/rocky/<8|9|10>/BaseOS/aarch64/os`. Only
  **Rocky 10** (6.12 el10) is in-window; Rocky 9 (5.14) and 8 (4.18) predate
  the trigger and are not affected. AlmaLinux is the leading indicator for
  the fixed el10 kernel version.
- **Amazon Linux** core (aarch64/Graviton): AL2023 default `kernel` is the
  6.1 stream (< 6.10, not affected); its opt-in `kernel6.12` / `kernel6.18`
  streams are in-window. AL2 `kernel` is 4.14 (not affected). Resolve the
  aarch64 `mirror.list` then read repodata; cross-check
  `alas.aws.amazon.com`.

## Debian kernel version source

Use the dak-backed madison API for the source-package version per suite
(unstable=sid, testing=forky, stable=trixie, oldstable=bookworm,
oldoldstable=bullseye); the version is arch-independent:

```
curl -fsSL 'https://api.ftp-master.debian.org/madison?package=linux&s=sid,forky,trixie,bookworm,bullseye&text=on'
```

Only suites at 6.10+ are affected: trixie (6.12) is in-window; forky/sid
(7.0/7.1) carry the fix; bookworm (6.1) and bullseye (5.10) are not
affected.

## Platform-specific notes

- **arm64 only:** the bug is in the GICv3/4 ITS emulation, which x86 KVM
  does not have. x86 hosts are entirely out of scope — that is Januscape's
  territory.
- **EL family:** only EL10 (6.12) is in-window on arm64; EL8/EL9 predate the
  trigger. AlmaLinux ships ahead of Rocky/RHEL.
- **Amazon Linux:** default AL2023 (6.1) and AL2 (4.14) predate the trigger
  → not affected; the opt-in 6.12/6.18 streams are in-window.
- **Proxmox VE:** x86-only product — not tracked here (see Januscape).
- **No runtime mitigation:** the vGIC-ITS path cannot be disabled without
  breaking guest interrupts, so a `:warning:` is only for a staged fix, and
  the only interim posture is not running untrusted arm64 guests.
